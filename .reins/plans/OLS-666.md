## Approach

Promote `TokenHandler` from a utility with scattered static helpers into the single
authority for all token measurement and budget-related counting. Today three concerns
leak into `docs_summarizer.py`: (1) tool-call JSON measurement uses `json.dumps` +
`_get_token_count`, (2) tool-result measurement uses raw `len()` (no buffer), and
(3) tool-definition measurement uses its own `json.dumps` + `_get_token_count` call.
Meanwhile `limit_conversation_history` uses `f"{message.type}: {message.content}"`
which silently drops `tool_calls` and ToolMessage metadata.

The fix adds three public methods to `TokenHandler` — `measure_message`,
`measure_tokens` (text→buffered count), and `measure_tool_definitions` — then
rewires all callers in `docs_summarizer.py` to use them. `limit_conversation_history`
switches to `measure_message`, which returns the same count as today for plain
messages and a more accurate count for tool-bearing messages.

This approach was chosen over introducing a `TokenBudget` class because the current
call sites only need consistent measurement, not stateful budget management; YAGNI
applies.

## Per-Criterion Plan

### AC 1: Budget partitioning, token measurement, budget tracking, and history truncation use a single shared source of truth

- **Approach:** Add a public `measure_tokens(text: str) -> int` method to
  `TokenHandler` that encapsulates the `_get_token_count(text_to_tokens(text))`
  pattern. All external callers that currently call `_get_token_count` directly or
  use `len(text_to_tokens(...))` switch to `measure_tokens`. This makes
  `TokenHandler` the only place where the buffer weight and encoding are applied.
  Also add `measure_tool_definitions(tools_json: str) -> int` as a thin wrapper so
  `docs_summarizer.py` no longer duplicates the `json.dumps` + count pattern.
- **Where:** `ols/utils/token_handler.py:TokenHandler` (new methods),
  `ols/src/query_helpers/docs_summarizer.py:DocsSummarizer._process_tool_calls_for_round`
  and `iterate_with_tools` (replace inline counting),
  `ols/src/query_helpers/docs_summarizer.py:DocsSummarizer._tool_result_chunk_for_message`
  (replace `len(content_tokens)` with `measure_tokens`).
- **Test:** Existing unit tests continue to pass. New unit test asserts
  `measure_tokens` matches `ceil(len(text_to_tokens(text)) * TOKEN_BUFFER_WEIGHT)`.

### AC 2: Token measurement correctly handles all LangChain message types including AIMessage with tool_calls and ToolMessage

- **Approach:** Add `measure_message(message: BaseMessage) -> int` to `TokenHandler`.
  It dispatches on message type:
  - `AIMessage` with non-empty `tool_calls`: count = buffered tokens of
    `f"{message.type}: {message.content}"` + buffered tokens of
    `json.dumps(message.tool_calls)`.
  - `ToolMessage`: count = buffered tokens of
    `f"tool ({message.name}): {message.content}"` (includes the tool name which is
    part of the wire format).
  - All other `BaseMessage` subclasses: count = buffered tokens of
    `f"{message.type}: {message.content}"` (preserves current behavior).
  Each message adds +1 for the newline separator (matching the existing convention).
  Update `limit_conversation_history` to call `self.measure_message(message)` instead
  of the inline `f"{message.type}: {message.content}"` formula.
- **Where:** `ols/utils/token_handler.py:TokenHandler.measure_message` (new),
  `ols/utils/token_handler.py:TokenHandler.limit_conversation_history` (update loop body).
- **Test:** New tests for `measure_message` with `HumanMessage`, `AIMessage` (content
  only), `AIMessage` (with `tool_calls`), `ToolMessage`. Verify that for simple
  messages (HumanMessage, AIMessage with content-only) the count equals the legacy
  formula. For AIMessage with `tool_calls`, verify the count is greater than the
  content-only count. For ToolMessage, verify the name is included.
  New test for `limit_conversation_history` with a mixed history containing
  `AIMessage(content="", tool_calls=[...])` and `ToolMessage(...)` to verify
  they are sized correctly and truncation works.

### AC 3: Tool output budget tracking and prompt-phase budget tracking are unified

- **Approach:** The root inconsistency is that `_tool_result_chunk_for_message` uses
  raw `len(content_tokens)` (no buffer) while all prompt-phase counting uses the
  buffered `_get_token_count`. Fix by replacing `len(content_tokens)` with
  `self.measure_tokens(str(tool_call_message.content))` (or the underlying buffered
  method via `token_handler`). Similarly, the AI tool-call message counting in
  `_process_tool_calls_for_round` currently calls `TokenHandler._get_token_count(
  token_handler.text_to_tokens(json.dumps(tool_calls)))` — replace with
  `token_handler.measure_tokens(json.dumps(tool_calls))`. Both paths now use the
  same buffered measurement, making the tool budget tracker consistent with the
  prompt-phase budget tracker.
- **Where:**
  `ols/src/query_helpers/docs_summarizer.py:DocsSummarizer._tool_result_chunk_for_message`
  (replace `len(content_tokens)` with `token_handler.measure_tokens(...)`),
  `ols/src/query_helpers/docs_summarizer.py:DocsSummarizer._process_tool_calls_for_round`
  (replace `TokenHandler._get_token_count(token_handler.text_to_tokens(...))` with
  `token_handler.measure_tokens(...)`),
  `ols/src/query_helpers/docs_summarizer.py:DocsSummarizer.iterate_with_tools`
  (replace tool-definitions counting with `token_handler.measure_tokens(...)`).
- **Test:** Existing tool-calling tests in `test_docs_summarizer.py` continue to pass.
  Verify that `_tool_result_chunk_for_message` now returns a buffered count by
  checking the returned token count against `measure_tokens` for the same content.

### AC 4: No behavioral regression — existing request flows produce equivalent results

- **Approach:** For messages without `tool_calls` (HumanMessage, AIMessage with
  content), `measure_message` produces the same value as the old formula
  `_get_token_count(text_to_tokens(f"{message.type}: {message.content}")) + 1`.
  This is ensured by implementation: the default branch in `measure_message` uses
  exactly that formula. All existing tests for `limit_conversation_history`,
  `calculate_and_check_available_tokens`, `truncate_rag_context`, and
  `truncate_tool_output` must pass unchanged. The only behavioral change is that
  tool-related messages now get a more accurate (typically larger) count, which is
  the desired fix. The tool budget change (unbuffered → buffered for tool results)
  makes tool tracking ~10% more conservative, which is within the safety margin the
  buffer was designed for and consistent with how all other counting works.
- **Where:** All test files — no test modification needed for regression tests; they
  should pass as-is.
- **Test:** Run the full unit test suite (`make test-unit`). Run linting and type
  checks (`make verify`). Existing `test_limit_conversation_history` tests with
  `HumanMessage`/`AIMessage` must produce identical results.

## Risk Assessment

- **Budget shift for tool results:** Switching from `len()` to buffered `_get_token_count`
  for tool-result content makes tool budget tracking ~10% more conservative. This
  could cause tools to be skipped earlier if the global `max_tokens_for_tools` budget
  is tight. Mitigation: the 10% buffer was always intended; the current raw counting
  was the inconsistency.
- **History truncation with tool messages:** If conversation history already contains
  `AIMessage` with `tool_calls`, the more accurate counting could truncate history
  more aggressively than before. Mitigation: this is the correct behavior (the old
  counting was under-counting), and OLS-2685 explicitly depends on this fix.
- **Test mocks that hard-code token counts:** Some tests in `test_docs_summarizer.py`
  mock `TokenHandler` or patch constants. These may need adjustment if the counting
  interface changes. Mitigation: keep the public method signature simple and backward
  compatible; `_get_token_count` remains available as an internal static method.
