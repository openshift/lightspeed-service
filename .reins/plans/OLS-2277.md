## Approach

Replace the existing character-level truncation in the tool execution
chain with an offload-to-disk mechanism. When a single tool's output
exceeds `max_tokens_per_tool_output` (default 8,000 tokens), write the
full content to a temporary file and inject a short placeholder into
the conversation. Two retrieval tools are conditionally registered so
the model can search and read offloaded content:

- **`search_offloaded_content`** â€” grep-style regex search with
  context lines. Use case: find where something is.
- **`read_offloaded_content`** â€” read a specific line range. Use
  case: examine a section found via search.

This mirrors the proven Cursor/Claude Code pattern: search to find,
read to examine. All files are deleted when the HTTP request completes
via a try/finally block in `generate_response`. Truncation is preserved
as a fallback when disk writes fail or content exceeds the 50 MB hard
cap.

This is a single coherent change: the offloading mechanism, the
retrieval tools, the config surface, and the lifecycle management
are all tightly coupled and meaningless in isolation.

## Per-Criterion Plan

### AC 1: Tool outputs exceeding threshold written to temp file

- **Approach:** Introduce an `OffloadManager` class in the new module
  that owns a `try_offload(text, tool_name)` method. This method
  estimates token count using the existing 4 chars/token heuristic,
  and if the output exceeds the threshold, writes the full text to a
  uniquely-named file under the configured storage directory. The
  method returns either the original text (under threshold or fallback)
  or a placeholder string. The ref_id is a UUID4 stored in an
  in-memory allowlist dict mapping ref_id to file path (ADR decision
  5 â€” never derive paths from LLM input).

  File creation uses `os.open` with `O_CREAT | O_EXCL` to prevent
  symlink-following attacks â€” the call fails if the target path
  already exists, eliminating the race window between check and
  write.

  The integration point is `execute_tool_call` in `tools.py`. After
  the raw tool output is obtained from the coroutine, but before
  text extraction, the offload manager (if present) checks the size.
  Concretely: `_extract_text_from_tool_output` is refactored to
  separate text conversion from size enforcement. A new wrapper in
  `execute_tool_call` calls the manager's `try_offload` on the
  converted text, falling through to the existing truncation logic
  on failure.

  The `OffloadManager` is passed as an optional parameter through
  the chain: `execute_tool_calls_stream` â†’
  `_execute_single_tool_call_stream` â†’ `_execute_with_retries` â†’
  `execute_tool_call`. All default to `None` so existing callers
  (including tests) are unaffected.

- **Where:** `ols/src/tools/offloaded_content.py:OffloadManager`,
  `ols/src/tools/tools.py:execute_tool_call`,
  `ols/src/tools/tools.py:_extract_text_from_tool_output`

- **Test:** Unit test that a tool output exceeding the threshold
  results in a file on disk and a placeholder returned instead of
  the full text. Test that outputs under the threshold pass through
  unchanged.

### AC 2: Placeholder message with ref_id, size, and instructions

- **Approach:** The placeholder is a short string (~50-100 tokens)
  built by `OffloadManager.try_offload`. It includes: the ref_id,
  the original tool name, total line count, byte size, and a clear
  instruction telling the model to use `search_offloaded_content`
  to find relevant information or `read_offloaded_content` to read
  specific line ranges. The wording follows patterns proven in
  Claude Code and Cursor (ADR decision 3).

- **Where:** `ols/src/tools/offloaded_content.py:OffloadManager.try_offload`

- **Test:** Assert the placeholder string contains the ref_id, line
  count, byte size, tool name, and retrieval instructions mentioning
  both tools. Assert it is under 100 tokens (using the existing
  token handler).

### AC 3: Search and read retrieval tools

- **Approach:** `OffloadManager.build_retrieval_tools` returns a
  list of two `StructuredTool` instances:

  **`search_offloaded_content(ref_id, pattern, context_lines)`:**
  Reads the offloaded file, compiles the pattern as a regex, finds
  matching lines, and returns windowed output with line numbers and
  context â€” identical to `grep -n -C N` format. Each match block
  is separated by `--`. A header line states the ref_id, pattern,
  and match summary. The tool's Pydantic args schema is defined as
  a nested class so models get a clean JSON schema.

  Regex execution is wrapped with a timeout to prevent ReDoS â€” a
  malicious or poorly-constructed pattern (e.g. `(a+)+$`) could
  cause catastrophic backtracking on large files. If the regex
  times out, the tool returns an error message suggesting the
  model simplify its pattern.

  **`read_offloaded_content(ref_id, start_line, end_line)`:**
  Reads the offloaded file and returns the requested line range
  with line numbers. Validates that start_line and end_line are
  within bounds (clamps to file length rather than erroring).
  Useful after a search locates a region of interest and the model
  needs wider context (e.g., a full stack trace, a complete YAML
  block).

  Both tools are created with `readOnlyHint: True` in their tool
  metadata annotations. This ensures they bypass the Human-in-the-
  Loop approval flow regardless of `approval_type` configuration â€”
  the tools only read local offloaded content and are inherently
  safe.

  The tools participate in the existing `iterate_with_tools` loop
  alongside MCP tools â€” no inner loop or sub-agent. Each retrieval
  call costs one round from `max_iterations`. The model decides
  whether to search, read, or answer based on the results. The
  placeholder wording should guide the model to search with specific
  patterns rather than exhaustively scanning.

  Retrieval tool outputs are never candidates for offloading
  themselves â€” `try_offload` skips tools whose names match the
  retrieval tools, preventing recursion.

  Both tools are registered into the tools list from
  `iterate_with_tools` in `docs_summarizer.py`. After each
  `_process_tool_calls_for_round` completes, if
  `offload_manager.has_offloaded_content` is True and the tools
  have not yet been added, they are appended to both `all_mcp_tools`
  (the list passed to `bind_tools` for the LLM) and `all_tools_dict`
  (the nameâ†’tool resolution map). This handles the mid-loop injection
  described in ADR decision 6 / constraint TC-012.

- **Where:** `ols/src/tools/offloaded_content.py:OffloadManager.build_retrieval_tools`,
  `ols/src/query_helpers/docs_summarizer.py:DocsSummarizer.iterate_with_tools`

- **Test:** Unit test the search function directly: verify line
  numbers are correct, context windows are the right size, the
  grep-style format is parseable. Test with multi-line files,
  patterns matching at file boundaries, and overlapping context
  windows that should be merged. Unit test the read function: verify
  correct line range returned, boundary clamping, line number
  formatting.

### AC 4: Model can call retrieval tools multiple times per file

- **Approach:** No per-ref call limit. Both retrieval tools are
  stateless â€” each call reads the file independently. The tools
  remain registered for the duration of the tool-calling loop.
  Since the offloaded file persists until request cleanup, any
  number of calls against the same ref_id are supported.

  Typical workflow: model searches for a pattern, gets match at
  line 500 with 3 lines of context, then reads lines 480-550 to
  get the full stack trace.

- **Where:** Inherent in the design; no specific code needed beyond
  not adding a call counter.

- **Test:** Unit test that calls search then read on the same
  ref_id, verifying each returns correct results. Test multiple
  search calls with different patterns.

### AC 5: Search results capped with total match count

- **Approach:** A constant (e.g. 50) limits the number of match
  blocks returned per search call. The search function counts all
  matches first, then returns only the first N with a header line:
  "Showing N of M total matches". If the model needs more, it can
  refine its pattern. This also naturally bounds the search tool's
  output size, keeping it under normal token budget enforcement
  (ADR decision 7).

  The read tool is naturally bounded by the line range requested.
  A reasonable upper bound on the range size (e.g., 500 lines)
  prevents the model from requesting the entire file via read.

- **Where:** `ols/src/tools/offloaded_content.py:OffloadManager`

- **Test:** Offload a file with many matches, verify only the cap
  number are returned, and the header correctly reports the total.
  Test read with a range exceeding the cap, verify it is clamped.

### AC 6: Offloaded files deleted on request completion

- **Approach:** `OffloadManager.cleanup` iterates the allowlist and
  deletes each file, logging any I/O errors but never raising.
  `DocsSummarizer.generate_response` wraps its body in try/finally,
  calling `offload_manager.cleanup()` in the finally block. Since
  `generate_response` is an async generator, the finally runs when
  the generator is exhausted or closed â€” covering both successful
  completion and mid-stream errors/cancellation. The storage
  directory itself is not deleted (it may be shared across
  concurrent requests).

  Additionally, a startup cleanup runs when the service starts:
  any files in `offload_storage_path` from a previous run (e.g.
  after a crash where the try/finally didn't execute) are deleted.
  This goes in the application startup path (e.g. `lifespan` or
  `on_startup` hook) â€” a single `shutil.rmtree` + `mkdir` of the
  storage directory.

- **Where:** `ols/src/tools/offloaded_content.py:OffloadManager.cleanup`,
  `ols/src/query_helpers/docs_summarizer.py:DocsSummarizer.generate_response`

- **Test:** Create an OffloadManager, offload content, call cleanup,
  verify files are gone. Also test that cleanup tolerates already-
  deleted files (idempotent). Integration-level: verify the finally
  block runs when the generator is closed early.

### AC 7: Truncation as fallback

- **Approach:** `OffloadManager.try_offload` returns the original
  text (not offloaded) in three cases: (a) the text is under the
  threshold, (b) the raw byte size exceeds 50 MB, (c) the disk
  write fails (permissions, disk full, any `OSError`). In cases
  (b) and (c) it logs a warning. When `try_offload` returns
  un-offloaded text, the existing truncation path in
  `_extract_text_from_tool_output` applies as it does today.

  When no `OffloadManager` is provided (e.g. offloading not
  configured or MCP servers absent), the code path is identical
  to today â€” pure truncation.

- **Where:** `ols/src/tools/offloaded_content.py:OffloadManager.try_offload`,
  `ols/src/tools/tools.py:execute_tool_call`

- **Test:** Test disk write failure (mock `open` to raise `OSError`)
  returns original text. Test 50 MB+ content returns original text.
  Both cases should log a warning.

### AC 8: Retrieval tools handle errors gracefully

- **Approach:** Both tools catch all exceptions and return
  descriptive error strings. Specific cases:

  **search_offloaded_content errors:**
  - Invalid ref_id (not in allowlist): "Error: unknown reference
    '{ref_id}'. Available references: [list]"
  - Bad regex (re.error): "Error: invalid search pattern
    '{pattern}': {error details}"
  - Regex timeout (ReDoS): "Error: search pattern '{pattern}'
    timed out. Try a simpler pattern."
  - No matches: "No matches found for pattern '{pattern}' in
    {ref_id} ({line_count} lines)"
  - I/O error reading file: "Error: could not read offloaded
    content for '{ref_id}': {error}"

  **read_offloaded_content errors:**
  - Invalid ref_id: same as search
  - I/O error reading file: same as search
  - Out-of-bounds lines: clamp silently rather than error (the
    model doesn't need to know the exact line count before reading)

  Neither function ever raises (ADR constraint TC-008). The model
  receives a helpful error message and can adjust its query.

- **Where:** `ols/src/tools/offloaded_content.py:OffloadManager`

- **Test:** Parametrized test covering each error case for both
  tools, verifying the returned string contains the expected
  diagnostic information and that no exception propagates.

### AC 9: Storage path configurable

- **Approach:** Add `offload_storage_path` to `OLSConfig` with
  default `/tmp/ols-offloaded/`. Add `max_tokens_per_tool_output`
  to `ModelParameters` with default 8,000. Add corresponding
  constants to `ols/constants.py`.

  The storage directory is created on first offload (mkdir -p
  equivalent), not at startup â€” avoids side effects when offloading
  is never triggered. Validation at config load time checks the
  path is a string (no file existence check since it's created
  on demand).

- **Where:** `ols/app/models/config.py:OLSConfig`,
  `ols/app/models/config.py:ModelParameters`,
  `ols/constants.py`

- **Test:** Config model tests: verify defaults, verify custom
  values are accepted, verify the fields appear in serialized
  config.

### AC 10: Works on streaming and non-streaming endpoints

- **Approach:** Both streaming and non-streaming paths consume
  `DocsSummarizer.generate_response`, which is the single place
  where the `OffloadManager` is created and cleaned up. The
  non-streaming path (`create_response` â†’ `drain_generate_response`)
  fully drains the async generator, triggering the finally block.
  The streaming path (`response_processing_wrapper`) also drains
  or closes the generator, triggering cleanup.

  The `streaming` flag is already threaded through to
  `execute_tool_calls_stream` (for approval flow); the offload
  manager is orthogonal to this flag.

  ADR constraint TC-013 notes that both paths share
  `_process_tool_calls_for_round` â€” the offloading integration
  is in this shared path, so it applies to both automatically.

- **Where:** No additional code for this AC â€” it's a consequence
  of placing the lifecycle in `generate_response`.

- **Test:** Existing streaming and non-streaming integration tests
  exercise both paths. Add a focused test that creates a
  DocsSummarizer with `streaming=True` and `streaming=False`,
  verifies offloading and cleanup happen in both cases.

## Risk Assessment

- **Breaking existing truncation behavior:** The truncation code
  path is preserved as a fallback and is exercised when offloading
  is not configured, fails, or for content under the threshold.
  Existing truncation tests should continue to pass with
  `offload_manager=None`.

- **Concurrent tool execution safety:** `execute_tool_calls_stream`
  merges parallel tool generators via `aiostream`. Multiple tools
  may call `try_offload` concurrently. Since Python's asyncio is
  single-threaded and `try_offload` performs synchronous file I/O
  (no `await` between dict mutation and file write), there are no
  race conditions. UUID4 file names prevent collisions.

- **Retrieval rounds consuming max_iterations budget:** Each
  retrieval call (search or read) costs one round of the
  tool-calling loop. With `DEFAULT_MAX_ITERATIONS=5` (ask mode),
  a model that offloads one tool result, searches it, then reads
  a section would consume 2 extra rounds. The ADR acknowledges
  this (TC-011). The existing configurable `max_iterations`
  mitigates this. Document as a tuning consideration in config
  examples.

- **Model compliance:** The model must recognize the placeholder
  and call the retrieval tools. If it doesn't, the answer is based
  on the placeholder alone â€” no worse than today's truncated output.
  The tool descriptions and placeholder wording should be tested
  empirically with the primary models (GPT-4o, Granite).

- **Security hardening:** Three defensive measures are built into
  the implementation: (1) `O_CREAT | O_EXCL` for file creation to
  prevent symlink attacks, (2) regex timeout on search to prevent
  ReDoS from malicious patterns, (3) startup cleanup of the storage
  directory to remove orphaned files from previous crashes. The
  ref_id allowlist (ADR decision 5) prevents path traversal.

## Resolved Questions

1. **Max matches per search call:** 50 (constant). Each match is a
   line plus context â€” 50 matches with 3 lines of context each is
   ~350 lines of output, well within token budget limits.

2. **Max line range for read_offloaded_content:** 500 lines
   (hardcoded constant). This is the upper bound on `end_line -
   start_line`. The model uses this to read a window around a match
   found by search â€” e.g., search finds an error at line 500, model
   reads lines 480-550. The cap prevents requesting the entire file
   while being generous enough for any reasonable context window.
