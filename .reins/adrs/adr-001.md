# ADR-001: Big Tool Result Offloading

| Field     | Value |
|-----------|-------|
| **Status**   | Proposed |
| **Date**     | 2026-03-24 |

## Context

When an MCP tool in OpenShift Lightspeed produces output that exceeds the context window capacity, OLS truncates it and appends a warning message indicating the data is incomplete. This is implemented in `ols/src/tools/tools.py` through a two-stage process:

- **Character-level truncation** in `_extract_text_from_tool_output()` — estimates tokens at 4 chars/token, cuts at newline boundaries
- **Token budget enforcement** in `enforce_tool_token_budget()` — proportionally scales or targets the longest message when combined tool outputs exceed `max_tokens_for_tools` (default 32,000)

**The problem:** truncation causes information loss. The model may miss critical details buried in the truncated portion — an error message at line 500 of a pod log, a misconfigured field deep in a YAML dump, a specific event in a long audit trail. The model has no way to know what it's missing or to ask for more.

This also applies to user-provided attachments (logs, YAML configs), which are currently inlined into the query string and consume context window space.

## Decision

Replace truncation with a **two-phase offload-and-retrieve workflow**:

1. **Offload:** When tool output exceeds the threshold, write the full content to a temporary file on disk. Inject a short placeholder (~50-100 tokens) into the conversation telling the model what was offloaded and how to retrieve it.
2. **Retrieve:** Provide the model with two retrieval tools:
   - **`search_offloaded_content`** — accepts a reference ID, search pattern (keyword or regex), and context window size. Returns matching lines with surrounding context in grep-style windowed format with line numbers. Use case: *find where something is.*
   - **`read_offloaded_content`** — accepts a reference ID, start line, and end line. Returns the exact line range with line numbers. Use case: *read a specific section* (e.g., a full stack trace found via search).

The model can call the retrieval tools multiple times with different patterns or line ranges to extract what it needs, then continue with the user's task. The end user sees the same conversation flow — the offloading is transparent.

## Options Considered

### Option 1: Smarter truncation (rejected)

Keep truncation but make it more intelligent — e.g., prioritize error lines, summarize before truncating, or truncate from the middle rather than the end.

**Why rejected:** Still loses information. Any heuristic about what to keep assumes we know what's relevant before the model has seen the data. The model is better positioned to decide what matters.

### Option 2: Offload + single multiplexed retrieval tool (refined away)

Original proposal: offload to disk and provide one retrieval tool with three operations — `read_lines` (by line range), `search` (by pattern), and `metadata` (file stats) — selected via an `operation` parameter.

**Why refined:** A multiplexed tool with conditionally required parameters creates a complex schema that some models handle poorly. Metadata can be included in the placeholder message. However, line-range reading is valuable *after* a search locates a region of interest. Refined to two separate tools with independent, simple schemas instead of one multiplexed tool.

### Option 3: Streaming chunks to the model (rejected)

Feed the tool output to the model in chunks across multiple turns, letting it process each chunk sequentially.

**Why rejected:** Consumes many LLM rounds (expensive), doesn't let the model randomly access the data, and creates a complex state machine for chunk management. The model also can't go back to re-read earlier chunks.

### Option 4: Offload + search-based retrieval, request-scoped (selected)

Write to disk, provide a grep-style search tool, delete files when the request completes. No TTL, no background cleanup, no cross-request persistence.

**Why selected:** Simplest approach that solves the problem. The model gets complete data access through targeted search. Request-scoped lifetime eliminates all cleanup complexity. Grep-style output is a proven pattern (Claude Code, Cursor) that models understand well.

## Key Design Decisions and Rationale

### 1. Offloading replaces the truncation code path

**Decision:** Offloading happens inside `execute_tool_call` / `_extract_text_from_tool_output`, replacing the existing truncation logic. Truncation is preserved only as a fallback if the disk write fails.

**Rationale:** This is the natural interception point — it's where the system already decides "this output is too large." Adding a parallel path would mean maintaining two size-checking mechanisms. Replacing it keeps the code simple and ensures all large outputs go through the same path.

**Alternative considered:** Intercept at `_process_tool_calls_for_round` level (after execution, before message insertion). Rejected because the raw output is already truncated by that point — we'd need to store it somewhere before truncation, adding complexity.

### 2. Request-scoped file lifetime (not conversation-scoped)

**Decision:** Offloaded files exist only during the processing of a single HTTP request. They are deleted in a `try/finally` block when the response is returned.

**Rationale:** The original proposal scoped files to the conversation lifetime with a configurable TTL. This introduced significant complexity:

- TTL management (background tasks or lazy cleanup)
- Conversation-end detection (when does a conversation "end" in a stateless HTTP API?)
- Orphaned file cleanup across pod restarts
- `conversation_id` threading through the tool execution chain (which doesn't carry it today)

None of this complexity is needed. The model only uses offloaded content during the tool-calling rounds of a single request. Once the response is generated, the offloaded content has served its purpose. Request-scoped lifetime eliminates all cleanup concerns — a `finally` block is sufficient.

### 3. Two retrieval tools: search + read (not a multiplexed tool, not search-only)

**Decision:** Two retrieval tools with independent schemas:

- **`search_offloaded_content(ref_id, pattern, context_lines)`** — regex search returning matching lines with N lines of surrounding context, like `grep -n -C N`. Use case: find where something is.
- **`read_offloaded_content(ref_id, start_line, end_line)`** — return exact line range with line numbers. Use case: read a specific section after search locates it.

**Rationale:**

This mirrors the proven tooling pattern used by Claude Code and Cursor, where AI models work with large files via **search to find, then read to examine**. A search hit with 3 lines of context often isn't enough — the model may need to read a 50-line stack trace, a full YAML block, or the surrounding function. Increasing `context_lines` globally is wasteful when only one match needs a wider view.

- **vs. multiplexed tool** (`read_lines`/`search`/`metadata` as an "operation" parameter): Complex schema with conditionally required params. Some LLM providers (especially Granite) handle this poorly. Two independent tools with simple schemas avoid this.
- **vs. three separate tools** (search + read + metadata): Metadata is included in the offload placeholder. Two tools are sufficient.
- **vs. search-only:** Search finds *where* something is, but the model often needs a wider view *around* the match. `read_offloaded_content` provides this without bloating every search result.
- **Token cost:** Two tool definitions consume ~60-80 definition tokens total — acceptable overhead given they are only registered when content is actually offloaded (Decision 6).

**Metadata as a separate tool:** Not needed — line count and size are included in the offload placeholder message. The model knows the total lines from the placeholder and can read any range.

### 4. `max_tokens_per_tool_output` becomes the offloading threshold

**Decision:** Repurpose the existing `max_tokens_per_tool_output` parameter (default 8,000 tokens) as the offloading trigger instead of introducing a new parameter.

**Rationale:** This parameter already answers the question "at what size is a tool output too large to handle inline?" Its current behavior (truncate at this threshold) is being replaced (offload at this threshold). Adding a separate `offload_threshold_tokens` would create two competing thresholds for the same concept. Reusing the existing parameter means zero config changes for operators — the default just works better.

### 5. In-memory allowlist for `ref_id` resolution (not path derivation)

**Decision:** The offload manager maintains a dict of `{ref_id: file_path}`. The retrieval tool looks up `ref_ids` in this dict. It never constructs file paths from the `ref_id` string.

**Rationale:** The retrieval tool accepts `ref_id` from the LLM. LLMs can be prompt-injected. If the manager derived a path like `storage_dir / ref_id + ".txt"`, a crafted `ref_id` (e.g., `../../etc/passwd`) could read arbitrary files. The allowlist pattern completely eliminates this attack vector — only UUIDs registered during the current request are resolvable. This aligns with the project's security posture (Bandit scanning, OWASP awareness).

### 6. Register retrieval tools only when content is offloaded

**Decision:** Both retrieval tools (`search_offloaded_content` and `read_offloaded_content`) are added to the tools list only when at least one output has been offloaded. They are not always-registered.

**Rationale:** Tool definitions consume tokens from the `max_tokens_for_tools` budget. The two retrieval tool schemas (~60-80 definition tokens combined) would be wasted on every request where no offloading occurs (the majority of requests). Conditional registration keeps the common case lean.

**Complication this creates:** For tool outputs, offloading happens inside the tool-calling loop. The retrieval tools may need to be added to the tools list mid-loop (after the first offload in a round). For attachments (Story 2), offloading happens before the loop. The implementation must handle both injection points.

### 7. Retrieval tool results subject to normal budget enforcement

**Decision:** Retrieval tool outputs are treated identically to MCP tool outputs for token budget purposes. No special exemptions from `enforce_tool_token_budget`.

**Rationale:** The alternative — exempting retrieval results from proportional truncation — creates a special case in the budget system. Since the search tool returns windowed results (matches + context), each call's output is naturally bounded and small. If the model requests a very broad search that returns many matches, the result should be subject to budget limits, just like any other large tool output. The model can refine its search pattern.

### 8. Fallback to truncation as a safety net

**Decision:** If the disk write fails (permissions, disk full, I/O error) or the content exceeds 50 MB, the system falls back to the existing truncation behavior and logs a warning.

**Rationale:** Offloading should never make the system less reliable than it is today. If something goes wrong with disk I/O, the user should still get an answer (albeit from truncated data, as they do today). The 50 MB hard cap prevents a single tool output from filling the ephemeral volume.

### 9. Single module for offload manager + retrieval tools

**Decision:** The offload manager and both retrieval tool definitions live in a single file: `ols/src/tools/offloaded_content.py`.

**Rationale:** They are tightly coupled — both retrieval tools must validate `ref_ids` against the manager's allowlist and access the stored content. Splitting them would create either a circular dependency or require an unnecessary abstraction layer. The components are not large enough to warrant separate files. This follows the project's KISS principle.

### 10. Attachment offloading as a separate story

**Decision:** Tool output offloading (Story 1) and attachment offloading (Story 2) are separate stories, with Story 2 depending on Story 1.

**Rationale:** They share the same offloading mechanism but have different integration challenges:

- Tool outputs are offloaded inside the tool-calling loop in `_process_tool_calls_for_round`
- Attachments are processed in `process_request()` before `DocsSummarizer` exists
- Attachment offloading requires enabling tool calling even without MCP servers (`_tool_calling_enabled` must be `True` if any content is offloaded)
- The offload manager must be created as a request-scoped object in `process_request()` and passed through to the summarizer

Story 1 is a complete, usable feature on its own. Story 2 extends it to a second input channel with its own integration concerns.

## Constraints Discovered During Code Analysis

These constraints were found by tracing the actual function signatures and code paths in lightspeed-service. They shaped the design decisions above.

| # | Constraint | Impact |
|---|-----------|--------|
| TC-001 | `execute_tool_call()` has no `conversation_id` parameter; full call chain doesn't propagate it | Led to request-scoped UUIDs instead of conversation-scoped file naming |
| TC-002 | `_tool_calling_enabled` is `False` when no MCP servers configured; attachments are processed before the summarizer exists | Attachment offloading (Story 2) must override this condition |
| TC-003 | Retrieval tool accepts `ref_id` from LLM (untrusted input) | In-memory allowlist required for security |
| TC-004 | `offload_storage_path` is service-wide; threshold is per-model | Config split between `OLSConfig` and `ModelParameters` |
| TC-007 | Retrieval tools could return too-large output if uncapped | Search results bounded by match cap + context window; read results bounded by line range limit; both subject to normal budget enforcement |
| TC-008 | Tool exceptions cause unrecoverable failures for the model | Both retrieval tools return error strings, never raise |
| TC-011 | Default `max_iterations=5` may be too few with retrieval rounds | Document as tuning consideration |
| TC-012 | Retrieval tools may need to be added to tools list mid-loop | Two injection points: pre-loop (attachments) and in-loop (tool outputs) |
| TC-013 | Streaming and non-streaming share `_process_tool_calls_for_round` | Must test both paths |

## Consequences

### Positive

- **No information loss:** The model has access to the complete tool output, regardless of size
- **Minimal context window consumption:** A 50,000-token tool output becomes a ~50-token placeholder
- **Transparent to users:** Same conversation flow, potentially better answers
- **Simple lifecycle:** Request-scoped files, no cleanup infrastructure
- **Secure:** In-memory allowlist prevents path traversal attacks
- **Graceful degradation:** Falls back to truncation if disk fails

### Negative

- **Additional LLM rounds:** The model needs 1-3 extra rounds to search and read offloaded content, increasing latency and token usage
- **Disk I/O:** Writes temporary files to disk (mitigated by ephemeral `/tmp` storage)
- **Model compliance:** Relies on the model actually calling the retrieval tool when it sees an offload notification (~90% expected compliance based on similar patterns in Claude Code/Cursor)
- **`max_iterations` pressure:** Retrieval rounds consume iteration budget, potentially limiting MCP tool rounds

### Risks

- Models that don't follow tool-use instructions well may ignore the offload notification and answer without the data. *Mitigation:* clear tool descriptions and placeholder wording; fallback is no worse than today's truncation.
- Very large files (near 50 MB) may have too many search results for the model to process effectively. *Mitigation:* match cap per search call; model can refine patterns.
- Pod restart during request loses offloaded files. *Mitigation:* request fails anyway on pod restart; no worse than today.
